## /docs 폴더를 만들지 않는 이유

이 저장소에 원본 문서(HWP, PDF, TXT 등)를 복사한 `/docs` 폴더를 만들지 않는다.

1. **원본이 이미 있다**: 보고서·참고자료는 `/Users/liberets/Downloads/NHI71/`에 있고 경로는 `CLAUDE.md`에 명시돼 있다. Claude는 새 세션에서도 `CLAUDE.md`를 읽는 것만으로 어느 파일을 참조해야 하는지 알 수 있다.

2. **저장소가 무거워진다**: 국정과제 TXT 하나만 312KB이고, 이전 기수 보고서 HWP/PDF는 훨씬 크다. git 이력에 대용량 바이너리가 쌓이면 clone·pull 속도가 크게 느려진다.

3. **이중관리 문제**: 원본 파일이 갱신될 때 `/docs` 복사본도 함께 업데이트해야 한다. 두 곳이 어긋나면 어느 쪽이 최신인지 판단할 수 없다.

4. **git의 용도가 아니다**: git은 텍스트 diff를 추적하는 도구다. 대용량 문서 파일은 변경 내역 추적이 의미 없고, 이진 파일은 diff도 볼 수 없다.

**대안**: 파일 경로와 구조 요약을 `CLAUDE.md`에 기록하고, 세부 내용이 필요할 때 `mcp__kordoc__parse_pages`로 원본을 직접 파싱한다.

## 재사용 가능한 프롬프트 모음

### HWP / PDF → TXT 변환

다음 경로에 있는 [.hwp / .pdf] 파일을 파싱해서 .txt로 변환해줘.

**입력 경로:** [절대경로]  
**출력 경로:** [절대경로] (없으면 생성해줘)  
**대상 파일:** 파일명에 "[키워드]"가 포함된 .hwp 및 .pdf 파일  
**제외 파일:** `._`로 시작하는 macOS 메타데이터 파일은 제외  

**특이사항:**
- 파일명이 한글이라 macOS NFD 인코딩 이슈가 있을 수 있으니 `unicodedata.normalize('NFC', ...)` 처리 필요
- HWPX 포맷(ZIP 기반)이 있을 수 있으니 magic byte로 판별해서 처리 (`PK\x03\x04`면 ZIP)
